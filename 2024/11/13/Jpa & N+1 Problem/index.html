Jpa & N+1 Problem

何謂N+1 Problem

N+1 問題通常出現在 ORM 框架 使用懶負載（Lazy Loading） 的情境下。

假設今天有兩張表 Student、Course，當你查詢Student這張表時，ORM 會執行一個主查詢（例如 SELECT * FROM Student），載入出所有的 Student ，但因為使用Lazy Loading的關係不會立即載入 Course這張表 。

當你遍歷每一個 Student 並嘗試存取他們的 Course 集合時，ORM 會為每個 Student 內的每筆資料執行一個新的查詢（例如 SELECT * FROM Course WHERE student_id = ?），因此總共需要 N + 1 次查詢：

1次查詢 用於載入所有 Student 記錄。
N次查詢 用於逐一載入每個 Student 對應的 Course 集合（其中 N 是 Student 的筆數）。

為什麼這會影響效能？

當 N 的數量較大時，每個 Student 都會觸發一次新的查詢來載入 Course，這會導致大量資料庫查詢，造成效能問題。特別是在擁有大量資料的系統中，N+1 問題可能會顯著拖慢程式!

如何解決N+1問題

我們將討論3種在JPA中解決N+1問題的方式:

  1. 將 FetchType 調整為 EAGER

  2. 使用@NamedEntityGraph、@EntityGraph Annotations

  3. 在JPQL中使用Join

在下面的範例中因為都是many to many，所以我們必須建立一張關聯student和course的表

將 FetchType 調整為 EAGER

直到JPA 2.0如果要關聯兩個Entity，我們除了使用FetchType.LAZY外，另外一個就是FetchType.EAGER。FetchType.EAGER的特性是無論關聯的Data是否有被使用，都會強制去載入另外一個關聯實體。

結論

雖然FetchType.EAGER看似很方便還能解決N+1問題，但是也是有不少隱藏的問題，

  1. 這個做法不夠彈性: 今天如果我們只想使用Student這個entity，但是在使用findAll()查詢所有的Student時，就會一併將我們不需要的courses資料都找出來。

  2. 可能拖慢應用程式效能或是消耗多餘伺服器資源: 今天當如果course這張table擁有上百萬筆資料，當我們查找Student時並生成course的實體時，就必須生成上百萬筆資料，我們試想今天如果Request一多，會造成多大的資源消耗，以及拖慢應用程式的效能。

@NamedEntityGraph 、@EntityGraph Annotations

在JPA2.1中，為了解決FetchType.LAZY及FetchType.EAGER都有缺點，@NamedEntityGraph、@EntityGraph這兩個Annotations誕生拉!

我們改寫一下Student Entity

  • @NamedEntityGraph
    定義了一個Entity Graph，用於指定在查詢 Student 實體時應一併載入的關聯實體

  • name = “Student.courses”
    指定該Entity Graph的名稱，方便在查詢時引用。這個名字可以是任意字串，在此例中它叫做 “Student.courses”，表示這是一個與 Student 相關的圖，並會載入 Student 實體的 courses 屬性。

  • attributeNodes = @NamedAttributeNode(“courses”)
    定義該Entity Graph中需要即時載入的屬性。 @NamedAttributeNode(“courses”) 表示在使用此實體圖表時，JPA 會載入 Student 實體的 courses 屬性。這表示在查詢 Student 時，關聯的 courses 集合會立即載入（通常使用 JOIN 查詢），而不是在存取 courses 時才進行懶載入。

  • @JoinTable
    註解用於定義一個中間表（即關聯式表），在資料庫中儲存多對多的關聯式對應。這裡指定的表名是 student_course，表示 Student 和 Course 之間的關聯關係。
    如果沒有 @JoinTable，JPA 會自動產生一個名字並不總是符合預期的中間表名，透過 @JoinTable 可以手動控制這個表名。

  • joinColumns
    定義 Student 在 student_course 表中的外鍵列。 @JoinColumn(name = “student_id”)

    表示 student_course 表中的 student_id 欄位是指向 Student 表主鍵 id 的外鍵。

  • inverseJoinColumns
    定義 Course 在 student_course 表中的外鍵列。 @JoinColumn(name = “course_id”)
    表示 student_course 表中的 course_id 欄位是指向 Course 表格主鍵 id 的外鍵。

結論

JPA 2.1 後加入的@NamedEntityGraph + @EntityGraph 就是要來解決原本FetchType.EAGER不夠彈性的問題，也是目前JPA官方比較推薦的做法!

但缺點就是當今天若有新增欄位或新增關聯時，必須將整個SQL改寫，增加了維運上的負擔。

總結

使用FetchType.EAGER、@NamedEntityGraph + @EntityGraph、JPQL都各有優缺點，使用的情境不同應該依情境選擇使用。FetchType.EAGER設定容易適合用在資料量不會很多的關聯上，@NamedEntityGraph + @EntityGraph設定上更為彈性，但當資料關聯性較為複雜時可能會較難設計，且學習成本較高，JPQL是很直覺的作法，SQL看起來就是一目了然，但是可能會造成後續維運上的困難。
