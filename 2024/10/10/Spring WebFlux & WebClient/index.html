Spring WebFlux & WebClient

前言

在 Spring 中，WebClient 和 RestTemplate 是兩個常見的 HTTP 客戶端工具。隨著 Spring 5 (Spring Boot 2.0.0)的引入，WebClient 成為了建議使用的非同步 HTTP 客戶端工具，而 RestTemplate 則是傳統的同步 HTTP 客戶端。

本篇將介紹WebClient ，以及比較WebClient 、 RestTemplate的優缺點，並且實作WebClient 的非同步作法與撰寫Unit Test。

Spring WebFlux介紹

Reactor 框架

Reactor是基於JVM的非阻塞API，它直接利用了JDK 8中的API，如CompletableFuture、Stream和Duration等。此外，Reactor還實現了Reactive Streams標準，確保了與其他響應式程式庫的兼容性。

Reactor 提供的主要特性：

  • 非阻塞編程：適合處理 I/O 密集型任務，如網路請求或DB I/O。

  • 異步數據流：通過 Mono 和 Flux，支持異步處理。

  • 豐富的運算子：提供轉換（map、flatMap）、過濾（filter）、錯誤處理（onErrorResume）、組合（concat、merge）等多種運算子來處理數據流。

而在Spring 5中引入的Spring WebFlux底層就使用了reactor-netty，而reactor-netty又引用了Reactor。進而與Spring MVC ，基於同步、阻塞請求的模型產生了互補的作用。

比較WebClient、RestTemplate

WebClient RestTemplate
優點 1. 支持非同步/非阻塞編程，適合處理高並發場景。
2. 作為 Spring 5 的新功能，WebClient 持續獲得優化。		 1. 簡單易用：同步編程模式，與一般的 Java 開發模式非常類似，開發者容易上手。
2. 廣泛使用：由於它是 Spring 中較早引入的工具，在很多項目中已被廣泛使用並且經過長期驗證。
缺點 1. 需要學習非同步、非阻塞與 Mono、Flux 等概念，學習曲線較長。 1. 無法非同步處理：RestTemplate 只支持同步請求，無法滿足高併發和需要異步處理的場景。
2. 性能瓶頸：同步阻塞導致大量 HTTP 請求時會阻塞線程，進而引發性能瓶頸。
3. 即將淘汰：Spring 官方已不再推薦 RestTemplate (不再進行優化)，建議新項目使用 WebClient。

3. subscribe()與block()

subscribe():

  • 屬於非同步、非阻塞的操作。

  • 當你調用 subscribe() 時，它會註冊一個回調函數來處理 Mono 或 Flux 中的結果，但不會等待結果。它允許應用繼續執行其他操作，並在數據流有結果後以非阻塞的方式處理它們。

  • 適合在需要異步處理數據流的場景中使用，比如在高併發應用中。

block():

  • 屬於同步、阻塞的操作。

  • 當你調用 block() 時，它會阻塞當前線程，直到 Mono 或 Flux 完成並返回結果。這類似於傳統的同步方法，它會等待結果返回，適合需要在當前線程中立即獲取結果的場景。

  • 不建議在需要高併發或反應式應用中頻繁使用，因為它會破壞非阻塞模型的優勢。

實作程式碼

同步、阻塞實作

使用block() 方法

block() 是一種同步操作，會阻塞當前線程，直到 Mono 或 Flux 完成並返回結果，或者發生錯誤。這個方法是從反應式流中提取數據的最直接方式，但它會使非阻塞的反應式流變成阻塞操作。

非同步、非阻塞實作

使用subscribe() 方法

subscribe() 是一個非同步操作，它不會阻塞線程，而是讓當前線程繼續執行。當數據流中的事件（值、錯誤、完成）發生時，subscribe() 提供的回調函數會被觸發。這使得應用程序可以在非阻塞的方式下處理數據，從而實現更高效的併發。
